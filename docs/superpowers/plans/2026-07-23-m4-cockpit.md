# M4 "Cockpit" Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Usability and safety hardening so a human can *watch* agents work and interject with confidence: close the read-only policy escape (`find -delete`), Cursor-parity edit flow (auto-apply + diff + revert), human-click adoption of subagent work (merge worktree, clean up), budget raise-and-resume, message attachments, and bounded prompt folding.

**Architecture:** All daemon changes ride existing seams: policy gets argument inspection for read-only commands; the shadow-git checkpoint store gains a diff view; a new human-only `/adopt` route commits + merges a child's worktree branch and removes the worktree; `/budget` raises a per-session cap and un-pauses; `POST /message` accepts inline attachments folded into the prompt inside the untrusted-data frame; the history fold gains a configurable event cap with an explicit truncation marker. Extension work is buttons and cards over those routes.

**Tech Stack:** Rust (axum, tokio, shadow-git via the existing checkpoint crate/module), TypeScript webview.

**Sealed-oracle contract (read first):** `tests/oracle/assert-m4-cockpit.sh` + `tests/oracle/stub-llm-m4.py` are authored, red-validated, and human-minted BEFORE this plan executes. **Never edit either file.** Build until `bash tests/oracle/assert-m4-cockpit.sh` prints `M4-COCKPIT-OK`.

### Task 6: `POST /sessions/{id}/adopt` — human-click merge of a subagent's work

**Files:**
- Modify: `daemon/composerd/src/api.rs`

**Interfaces:**
- `POST /sessions/{id}/adopt` with body `{"child": "<child-session-id>"}` — `{id}` is the PARENT (orchestrator) session. Authed. Semantics, in order:
  1. Validate: child exists, `kind == "subagent"`, `parent == {id}`, has a worktree. Else 4xx with a reason.
  2. In the child's worktree: if dirty, `git add -A` + commit as `composer` (`user.name=composer`, `user.email=composer@forge`, message `adopt <child>: subagent work`).
  3. In the parent workspace: `git merge --no-ff -m "adopt <child> (<title>)" fc/<child-lowercase>` . On merge conflict: abort the merge (`git merge --abort`), return 409 `{"error":"merge conflict", ...}`, leave the worktree in place.
  4. On success: `git worktree remove --force <worktree>` (via the existing `tools::worktree::remove`), delete branch `fc/<child>` (`git branch -d`, `-D` fallback), append an `adopt` event on the PARENT ledger: `actor:"human"`, `provenance:"trusted"`, body `{"child": <id>, "branch": "fc/<child>", "merge_commit": "<sha>"}`.
  5. Response `200 {"merge_commit": "<sha>"}`.
- This is a HUMAN action: no model turn is triggered; the adopt event folds into the parent's next prompt via the existing interdiction-visibility fold (verify it is picked up by the same loop-top fold that carries steer/inject — if that fold filters kinds, add `adopt` to the folded set).

- [ ] **Step 1: The failing check is the sealed oracle's anchor E** (dispatch a child that edits a file in its worktree; adopt; assert the file exists in the parent workspace, the worktree dir is gone, an `adopt` event with a real `merge_commit` is on the parent ledger, and 401 without bearer).
- [ ] **Step 2: Implement.**
- [ ] **Step 3: Verify** — anchor E; M2 oracle still green (its worktrees are never adopted — unchanged path).
- [ ] **Step 4: Commit** — `git commit -m "feat(api): human-only adopt route — merge subagent worktree, clean up, ledger the adoption"`

## Self-Review Notes (already applied)

- **Sealed-oracle compatibility:** the M1 oracle pins `auto_approve_edits` default `false` — parity ships via config template, not a default flip. The M2 oracle's worktrees are never adopted, so `/adopt` adds a path without changing one.
- **No invisible interventions:** adoption and budget raises are human actions on the ledger (`actor:"human"`), folded into the next prompt like steers. The model cannot merge into the user's branch — there is deliberately no tool for it.
- **Attachment injection surface:** attachment content enters prompts only inside the same untrusted-data frame as tool output; a poisoned attached file cannot smuggle instructions past §8.2 framing any more than a subagent report can.
- **Merge conflicts fail loudly and safely:** abort + 409 + worktree left intact; nothing half-merged.
- **YAGNI held:** no attachment persistence dedup, no diff-per-file API (one patch blob), no budget schedules, no follow-mode scroll setting (streaming + cards already give live sight).
