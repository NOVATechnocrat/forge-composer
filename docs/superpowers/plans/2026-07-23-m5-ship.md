# M5 "Ship" Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship-readiness: a model selector (choose which configured role/model a session runs as, switch mid-session — ledgered), model visibility (which model actually answered, everywhere costs show), a context gauge (prompt tokens vs the model's window), workspace rules (`AGENTS.md` folded into the system prompt, visibly), a non-interactive `composerd init` bootstrap, and a packaging script producing a distributable VSIX + release daemon.

**Architecture:** Roles are already named seats in config (`[roles.*]` = provider+model+escalation); the selector is UI over a new `GET /roles` + `POST /sessions/{id}/role` (ledgered `role_switch` event) + an optional `role` at session create. Visibility rides the existing usage/detail plumbing (`model` on usage events and `/sessions/detail`). The context gauge's denominator is a new optional `context_window` on the pricing table entries. Workspace rules are read at prompt-assembly time from the session jail root. `composerd init` writes a starter config; `scripts/package.sh` is authored but NOT oracle-gated (it needs vsce/network — the Architect runs it).

**Tech Stack:** Rust (composerd), TypeScript webview, bash packaging script.

**Sealed-oracle contract (read first):** `tests/oracle/assert-m5-ship.sh` + `tests/oracle/stub-llm-m5.py` are authored, red-validated, and human-minted BEFORE this plan executes. **Never edit either file.** Build until `bash tests/oracle/assert-m5-ship.sh` prints `M5-SHIP-OK`.

## Global Constraints

- **No invisible interventions:** a mid-session role/model switch MUST appear on the ledger (`role_switch`, `actor:"human"`) before it takes effect; the next turn uses the new role.
- Workspace rules are *data from the workspace*: fold `AGENTS.md` into the SYSTEM prompt under an explicit label (rules are house instructions, deliberately trusted — the human placed the file), but ledger a `rules` event so the human can see what the agents were told.
- Do not change sealed surfaces: existing event bodies keep their existing fields (adding NEW fields like `model` to usage bodies is fine — sealed oracles assert presence, not absence), existing tool schemas, `GET /sessions` shape.
- No new crate dependencies. `composerd init` and `--version` use the existing hand-rolled arg handling in `main.rs` (no clap).
- Commit after every task with the exact message given. Another agent may work in `extension/` concurrently — `git add` only your own files; retry once on index.lock contention.

## Self-Review Notes (already applied)

- **Selector = roles, not raw models:** picking from `[roles.*]` keeps escalation chains, pricing, and provider auth attached to the choice; a raw model dropdown would bypass all three. "Which model onboards as architect" is exactly a role row in config.
- **Switches are ledgered before they act** (`role_switch`, actor:human) — same doctrine as steer/adopt/budget: nothing about a session's behavior changes without a ledger trace the parent, the human, and the next prompt can see.
- **Rules are trusted on purpose, but visible:** AGENTS.md goes in the system prompt (house rules are instructions, unlike tool output), and the `rules` event keeps it auditable. The 32 KiB cap bounds prompt bloat; the context gauge makes the cost of big rule files visible.
- **Packaging is reviewed-not-proven** in the hermetic gate: vsce needs network, and a fake-VSIX assertion would be rubber-stamping. `composerd init` — the part users hit first — IS oracle-gated.
- **YAGNI held:** no per-message model override (per-session is the Cursor-parity unit), no rules hot-reload events on every change (byte-count re-emit only), no context-window auto-compaction (the M4 fold cap + gauge make overflow visible and bounded; smarter compaction is post-ship).
